# build-rite

Claude Code harness. Copied as `.claude/` into project repos via `sync.sh`.
Project-specific context → project-root `CLAUDE.md` (generated by `/br-discover`)

## Interaction rules

- Questions: 2-3 per round, max 7-8 per session. Format: `1. Question? A) B) C) D) Other:`
- Write discipline: confirm summary → explicit approval → atomic writes
- Style: bullets > prose · tables > lists · no filler

## Input parsing (shared across commands)

`$ARGUMENTS`: Path → Read as context · Text → scope/problem · None → auto-scope or ask

## Workflow

- CLAUDE.md is the knowledge container — `/br-discover` generates it; update after significant refactors
- Issues are the task system — batch related issues into PRs
- Refactor in chains — small focused steps, each its own commit
- `/br-plan` (read-only) → `/br-impl` (branch, code, verify, PR)
- `/br-review` before merge — severity-tiered, no auto-edits
- `/br-health` periodic project review — findings → docs + issues
- **Model hint:** sonnet works for discover/plan/impl · opus recommended for review/health

## Enforcement

- `git add <explicit files>` only — never `git add .` or `-A`
- Correct package manager (enforced by hook)
- No writes without user confirmation
- No build-rite attribution in any output — never expose `br-`, `build-rite`, or harness internals in text shown to users, commit messages, PR titles/bodies, or ticket comments. Use generic language (`/plan`, `/discover`, etc.) when referencing commands in user-facing output.
